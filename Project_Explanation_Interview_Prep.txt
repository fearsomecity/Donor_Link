# DonorNet Project - Interview Explanation Guide

This guide breaks down the full-stack architecture of the DonorNet platform, detailing its frontend, backend microservices, and how they communicate. Use this to prepare for your interviews and confidently explain the system.

## 1. High-Level Architecture
DonorNet is built using the **MERN Stack** (MongoDB, Express.js, React, Node.js) but follows a **Microservices Architecture**. Instead of having a single monolithic backend, the business logic is distributed across independent, specialized services that communicate with the frontend.

---

## 2. Frontend Implementation
The frontend is built for high performance, modern UI, and state reliability.

**Tech Stack:**
- **Framework:** React.js powered by Vite (for faster builds and hot reloading).
- **Styling:** Tailwind CSS (utility-first CSS framework for a responsive, dark-mode premium aesthetic).
- **State Management:** Zustand (a lightweight state management library used for managing authentication state globally in `authStore.js`).
- **Routing:** React Router v7 (`react-router-dom`) for handling client-side navigation.
- **Data Fetching:** Axios is used with an `apiClient.js` utility for making HTTP requests to the backend.
- **Icons & Charts:** Lucide-React for clean UI icons, Recharts for data visualization on dashboards.

**Key Pages (Located in `src/pages`):**
- **Auth:** `Login.jsx`, `RegisterDonor.jsx`, `RegisterHospital.jsx`.
- **Dashboards:** `DonorDashboard.jsx` (for donors), `HospitalDashboard.jsx` (for hospitals).
- **Features:** 
  - `AIAssistant.jsx` (AI-powered help utilizing Gemini AI).
  - `BookAppointment.jsx`, `DonorEligibility.jsx`, `DonorNeeds.jsx` (Donor-side tools).
  - `HospitalDonate.jsx`, `HospitalInventory.jsx`, `HospitalRequests.jsx` (Hospital-side tools).

**Reusable Components (Located in `src/components`):**
- `Navbar.jsx`, `Footer.jsx` for global layout.
- `ProtectedRoute.jsx` to restrict access to authenticated users based on their role (Donor vs Hospital).

---

## 3. Backend Implementation
The backend leverages a **Microservices Architecture** to ensure scalability, fault tolerance, and independent deployment of features.

**Tech Stack:**
- **Environment:** Node.js.
- **Framework:** Express.js for routing and handling HTTP requests.
- **Database:** MongoDB via Mongoose ORM for structured but flexible schemas.
- **Authentication:** JWT (JSON Web Tokens) and bcryptjs for password hashing.
- **AI Integration:** `@google/generative-ai` for powering the AI assistant.

**Microservices Architecture:**
The backend is split into multiple independent services (located in `backend/`):
1. **API Gateway (`api-gateway`):** Acts as the single entry point for the frontend. It routes incoming client requests to the appropriate individual microservice.
2. **Auth Service (`auth-service`):** Manages user registration, login, and token generation/validation.
3. **Donor Service (`donor-service`):** Manages specific actions related to donors (e.g., appointments, eligibility).
4. **Hospital Service (`hospital-service`):** Manages hospital profiles, inventory tracking, and donations.
5. **Request Service (`request-service`):** Manages blood requests bridging donors and hospitals.

**How the Services are Run:**
We use `concurrently` (defined in the root `package.json`) to spin up all microservices and the Vite frontend simultaneously with a single `npm start` command.

---

## 4. How Frontend Connects to Backend
1. **API Client:** In the frontend (`src/utils/apiClient.js`), an Axios instance is configured to define base URLs. 
2. **Request Flow:** When a user interacts with the UI (e.g., clicking 'Login'):
   - The React component calls a function in the Zustand store (`authStore.js`) or an API handler.
   - Axios sends an HTTP GET/POST request (along with JWT tokens if authorized) to the **API Gateway**.
3. **Routing in Gateway:** The API Gateway receives the request and forwards it to the respective microservice (e.g., `/api/auth/login` is routed to the Auth Service).
4. **Database Interaction:** The microservice processes the request, communicates with MongoDB using Mongoose, and sends back a JSON response.
5. **State Update:** The frontend receives the JSON response, updates its Zustand state or component state, and React re-renders the UI to reflect changes (like redirecting to the dashboard).

## 5. Potential Interview Questions & Answers

**Q: Why did you choose a Microservices architecture over a Monolith?**
*A:* Microservices allow us to scale individual parts of the app independently. For example, if the blood request feature gets heavy traffic, we can scale just the `request-service` without duplicating the entire backend. It also prevents a single point of failure and allows for easier maintenance.

**Q: How do you manage global state in the frontend?**
*A:* We use Zustand. I preferred it over Redux because it requires much less boilerplate, has a simpler API, and handles our authentication and user state perfectly without heavy setup.

**Q: How do you secure data and routes?**
*A:* On the backend, passwords are hashed with `bcryptjs`, and we use JWT (JSON Web Tokens) for stateless authentication. In the frontend, the `ProtectedRoute` component intercepts navigation and checks the Zustand store for valid authentication tokens before allowing access to dashboards and critical features.

**Q: How is the AI Assistant integrated?**
*A:* We integrated the Google Generative AI API (Gemini) into the backend. The frontend sends the user's prompt to our backend AI route, which securely communicates with Google's API using our hidden API Key, processes the response, and sends the text back to the frontend to simulate a chatbot experience using the `AIAssistant` component.
